Проект: Система структурированной обратной связи «Сотрудник → Сотрудник»
1.	Описание задачи
Необходимо разработать прототип веб-приложения, которое позволяет сотрудникам фиксировать опыт взаимодействия с коллегами в рамках рабочих эпизодов (задач, проектов, событий).
В компаниях такая обратная связь часто хранится разрозненно (переписки, устные оценки, полугодовые опросы без контекста). При этом именно качество взаимодействия, надёжность и влияние на команду напрямую влияют на управленческие и кадровые решения.
Цель системы — аккуратно и структурированно собирать обратную связь, хранить её в базе данных и предоставлять руководителю агрегированные отчёты и инфографику за выбранный период (включая полугодовой срез), с возможностью анализа динамики и категорий взаимодействия.
CRM компании на этапе прототипа недоступна, поэтому решение должно работать автономно, но быть архитектурно готовым к дальнейшей интеграции.
⸻
2.	Цель
Создать прототип системы, которая:
• собирает структурированную обратную связь «сотрудник → сотрудник»;
• хранит данные в БД;
• формирует отчёты и инфографику для руководителя/HR за выбранный период;
• соблюдает анонимность автора отзыва;
• реализует модель ролевого доступа.
Формат — веб-приложение с адаптивной версткой (desktop в приоритете).
⸻
3.	Основные требования
3.1 Структура отзыва
Отзыв должен содержать три обязательных элемента: 1. Контекст — описание рабочего эпизода (что произошло). 2. Оценка — позитив / негатив (возможно числовое значение). 3. Комментарий — краткое пояснение «почему» (в свободной форме).
Дополнительно:
• выбор одной или нескольких подкатегорий качества взаимодействия;
• логика ограничения повторных отзывов по одному и тому же эпизоду.
⸻
3.2 Анонимность и безопасность
• Автор отзыва анонимен для других сотрудников и руководителей.
• Руководитель/HR видит:
• агрегированные отчёты,
• инфографику за период,
• список отзывов по сотруднику/команде (без раскрытия автора).
• Обычные сотрудники не имеют доступа:
• к рейтингам,
• к сравнению сотрудников,
• к чужим комментариям,
• к отчётам по коллективу.
⸻
3.3 Полугодовой «срез»
• Система должна поддерживать отчёт за выбранный диапазон дат.
• Должен быть реализован типовой сценарий «полугодовой срез» (последние 6 месяцев).
⸻
3.4 Отчётность
Руководитель должен иметь возможность:
• получать сводку по сотрудникам;
• видеть баланс позитив/негатив;
• анализировать частоту подкатегорий;
• отслеживать динамику во времени;
• фильтровать по команде / подразделению / сотруднику;
• выгружать результаты в файл (агрегированно или обезличенно).
⸻
3.5 Управление категориями
Руководитель/модератор должен иметь возможность:
• добавлять новые подкатегории;
• скрывать или переименовывать существующие;
• адаптировать категории под текущие управленческие задачи.
⸻
4.	Сценарии использования
Сценарий 1. Позитивный отзыв
Сотрудник:
• выбирает коллегу;
• указывает рабочий эпизод;
• ставит позитивную оценку;
• выбирает 1–2 категории;
• оставляет короткий комментарий.
Система:
• сохраняет отзыв;
• применяет правила ограничения повторов.
⸻
Сценарий 2. Негативный отзыв (конструктивный формат)
Сотрудник:
• фиксирует негативный эпизод;
• описывает факт и последствия без личных оценок.
Система:
• проверяет корректность структуры;
• учитывает лимиты;
• делает отзыв доступным только руководителю/HR в отчётности.
⸻
Сценарий 3. Ограничение повторных отзывов
Пользователь пытается оставить второй отзыв по тому же эпизоду.
Система:
• не создаёт дубликат;
• либо блокирует повтор,
• либо предлагает обновить существующую запись.
⸻
Сценарий 4. Управление категориями
Руководитель/модератор:
• добавляет новые категории;
• изменяет или скрывает существующие.
Система:
• поддерживает гибкость и расширяемость классификации.
⸻
Сценарий 5. Отчёт за период
Руководитель:
• выбирает период (включая полугодовой срез);
• получает агрегированную аналитику:
• баланс позитив/негатив,
• распределение категорий,
• динамику,
• детализацию по сотруднику.
⸻
Сценарий 6. Экспорт данных
Руководитель:
• выгружает отчёт в файл (агрегировано или обезличенно).
Система:
• формирует структуру данных, пригодную для дальнейшего использования в корпоративной среде.
⸻
5.	Технические ограничения
• Формат решения — веб-приложение.
• Репозиторий с исходным кодом и инструкцией по запуску.
• Рабочий прототип должен быть доступен из открытой сети.
• Решение должно быть готово к дальнейшей интеграции с корпоративной C
